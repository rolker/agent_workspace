## Survey Summary

Overstory is a multi-agent orchestration system that turns a single Claude
Code session into a coordinator for worker agents spawned in git worktrees
via tmux, with inter-agent messaging over a custom SQLite "mail system" and
**tiered conflict resolution** on merge.

**Org:** `@os-eco` (via jayminwest) | **Stack:** Bun + TypeScript (strict),
Biome, Commander.js, `bun:sqlite` | **License:** MIT | **Cadence:** Lower
velocity than agent-orchestrator (8 merged PRs/month); issue load mostly
bug fixes and agent-role refinements

### Scope (from README + CLAUDE.md)

- `ov init` → `ov hooks install` → `ov coordinator start` in a project
- Supports **11 runtimes** via pluggable `AgentRuntime` interface: Claude
  Code, Pi, Codex, Gemini CLI, Copilot, Cursor CLI, Sapling, OpenCode,
  Aider, Goose, Amp
- Mail system mediates agent↔agent communication; coordinator checks mail
  via hook injection on `UserPromptSubmit`
- **"Your Claude Code session IS the orchestrator"** — no separate
  daemon; CLAUDE.md + hooks + `ov` CLI provide everything

### Architecture (CLAUDE.md)

**Hook-integrated orchestrator model:**
- `SessionStart` hook → runs `ov prime` (loads config, recent activity,
  mulch expertise)
- `UserPromptSubmit` hook → runs `ov mail check --inject` (surfaces new
  agent messages into the orchestrator's next prompt)
- You drive via `ov` CLI: `sling` spawns workers, `status` checks fleet,
  `nudge` wakes a stalled agent, `dashboard` for live TUI

**Base + Overlay prompt architecture:**
- **Layer 1 (Base)**: Reusable `agents/*.md` files defining role HOW
  (workflow, constraints, capabilities) — builder, coordinator, lead,
  merger, monitor, orchestrator, reviewer, scout, supervisor
- **Layer 2 (Overlay)**: Per-task `CLAUDE.md` generated by `ov sling`,
  written into the worker's worktree, defining WHAT (task ID, file
  scope, spec path, branch name)
- Orchestrator passes WHAT; the base defines HOW

**Hierarchical delegation:**
```
Orchestrator (your Claude Code session)
  └─ Team Lead (Claude Code in tmux, can spawn sub-workers)
       └─ Specialist Workers (Claude Code in tmux, leaf nodes)
```

### Runtime Abstraction (`docs/runtime-abstraction.md`)

Design document for decoupling from Claude Code. Key principles:
- **Extract, don't rewrite**: Phase 0 moves Claude behavior behind an
  interface with zero behavior change; new runtimes come after
- **Config-driven adapters**: most differences are flag names and file
  paths, not logic — adapters are mostly declarative
- **Incremental adoption**: a project can run Claude agents alongside
  Codex agents in the same swarm (runtime is per-agent, not per-project)
- **Headless-first**: runtimes supporting JSON/headless output are
  first-class; interactive-only are lower priority

35 coupling points collapsed into 4 abstraction surfaces: **Spawn**,
**Readiness**, **Hooks**, **Overlay + Transcripts**. Each new runtime =
~200–400 LOC adapter.

### STEELMAN.md — risk analysis against their own product

Unusual and valuable: the maintainers ship a **"Steelman Arguments Against
Agent Swarms"** document listing real failure modes before adopting
Overstory:

1. **Compounding error rates** at integration boundaries
2. **Cost amplification without proportional value** — "20-agent swarm
   over 6 hours: 8M tokens ($60). Single agent over 8 hours: 1.2M tokens
   ($9). The 2-hour speedup cost $51 in coordination overhead."
3. **Loss of coherent reasoning** — inconsistent naming, duplicated
   utilities, conflicting architectural assumptions, impedance mismatches
4. **Debugging becomes forensics** — traditional tools assume linear
   history; multi-worktree + mail + non-deterministic timelines make bug
   diagnosis expensive
5. **Premature decomposition** — swarms encourage breaking problems before
   they're understood
6. **Merge conflicts are the normal case** — tiered merge (up to AI
   resolver) still requires human review for semantic conflicts
7. **Infrastructure complexity** — tmux, worktrees, SQLite mail, watchdog
   daemons, hooks, dashboard — each a failure mode
8. **False sense of productivity** (read for full argument)

This is essentially the same reasoning behind our **Row 6 "Defer Tier 3
orchestration"** decision. Validation from the other side — Overstory's
own maintainers advise against deploying casually.

## Pending Review — candidate patterns worth triaging

### Portable concepts (high signal, implementable)

1. **STEELMAN document pattern** — maintain our own `docs/STEELMAN.md` or
   embed in relevant ADRs. Before we adopt any workspace-wide approach
   (coordinator, orchestration, new skill class), write the strongest
   case AGAINST it. Overstory's honest self-critique is a model.
   *Directly applicable now* — e.g., pair with our Tier 3 deferral
   and the coordinator-simmering decision to document the rejected paths
2. **Base + Overlay prompt architecture** — agent role definitions as
   reusable base markdown + per-task overlay. Our skills (plan-task,
   review-code) are single-file, mixing HOW and WHAT. A role catalog
   + per-invocation overlay could clarify our current patterns. Aligns
   with **Row 4 Opus 4.7 prompt audit** — we could emerge with a
   cleaner Base/Overlay split as part of that sweep
3. **Hook-integrated orchestrator** (no separate daemon) — "your Claude
   Code session IS the orchestrator." Matches our CLI-first /
   additive-only constraint perfectly. A coordinator skill (Row 7 D5
   simmering) could use the same pattern: hooks prime context and inject
   updates into YOUR session, not an external daemon
4. **Agent role catalog** (`agents/*.md`) — builder / coordinator / lead
   / merger / monitor / orchestrator / reviewer / scout / supervisor.
   If we ever want to formalize the "swarm-of-personas for /review-code"
   pattern (extracted from Row 7), these roles are a working vocabulary
   to start from

### Architectural references (understand, don't port)

5. **SQLite-backed mail system** — overkill for single-user,
   single-machine. File-based (we use progress.md) is simpler. Noted
   for record.
6. **Tiered merge resolution** (text → semantic → AI) — mature concept;
   we use manual review via PR + Copilot. Wouldn't adopt their tech
   but the tiering framework is a useful mental model
7. **Watchdog daemons** — PID file management, restart loops. The
   file-polling helper we extracted from Row 7 is the minimal form;
   Overstory's version is the full operational form
8. **Runtime abstraction (11 runtimes via adapters)** — validates the
   "framework resilience" principle we surfaced. Their 4-surface
   abstraction (Spawn/Readiness/Hooks/Overlay+Transcripts) is a useful
   framework. We may not ever need it ourselves, but it's the
   destination-state if we did

### Out of scope (skip)

- Actual swarm deployment — STEELMAN's own warnings + our Tier 2
  preference + D5 visibility constraint
- SQLite mail infrastructure — progress.md + direct observation covers
  us
- Multi-runtime adapter infrastructure — our adapter-file approach
  covers the same need with lower ceremony
- Dashboard TUI — CLI-first
- Beads / mulch / sd tooling — external CLIs Overstory uses; not for
  our stack
